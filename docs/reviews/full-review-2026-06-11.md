## 1. Баги и уязвимости

### HIGH

**H1. Гонка при параллельном refresh → случайные разлогины.**
`apps/web/src/lib/api-client.ts:182` — `refreshAccessToken()` не имеет
single-flight-дедупликации. Когда access-токен истекает (15 мин) и страница
делает несколько параллельных запросов, все получают 401 и каждый вызывает
`POST /auth/refresh`. Ротация (`tokens.service.ts:63`) отзывает старый refresh
после первого вызова → второй параллельный refresh получает 401 →
`useAuthStore.getState().clear()` → пользователь выброшен на login посреди
тренировки.
*Фикс:* хранить промис текущего refresh в модульной переменной; все конкурентные
вызовы ждут один и тот же промис.

**H2. Учётные данные закоммичены в репозиторий.**
`milena-create-log.txt` (корень репо) содержит логин/пароль
`mboiko@gmail.com / NfJmw9Pc2MlvvLoT`. Аналогично `statistics/*/IMPORT_REPORT.md`
(вне git, но на диске). Если это прод/общий инстанс — сменить пароль, файл
удалить из репо и истории, паттерн `*-create-log.txt` добавить в `.gitignore`.

### MEDIUM

**M1. Списки жёстко обрезаны без пагинации — данные молча пропадают.**
`matches.service.ts:39` (`take: 50`), `training.service.ts:56` (`take: 50`),
`calendar.service.ts:55,112,181`. У игрока с импортом из CueTracker матчей
легко станет больше 50 — старые просто исчезнут из UI (клиентская пагинация на
`/matches` листает только то, что пришло). *Фикс:* серверная cursor-пагинация
(`?cursor=&limit=`) + «показать ещё» на клиенте.

**M2. Глобальный body-limit 15 MB на все эндпоинты.**
`apps/api/src/main.ts:11-16` — лимит нужен только
`POST /drill-templates/recognize-layout`. Сейчас любой авторизованный (и
неавторизованный — парсинг происходит до guard'ов) может слать 15-мегабайтные
JSON на любой роут. *Фикс:* дефолтный лимит 256 KB, увеличенный — только на
роут распознавания.

**M3. Swagger открыт в проде.**
`main.ts:43-50` — `/api/docs` доступен публично на snooker.appshub.pl: полная
карта API для атакующего. *Фикс:* отключить при `NODE_ENV=production` или
закрыть basic-auth на nginx.

**M4. Нет детекта повторного использования refresh-токена.**
`tokens.service.ts:47-71` — использование уже отозванного токена возвращает
просто 401. Best practice (и сигнал кражи cookie): при reuse отзывать **все**
refresh-токены пользователя.

**M5. Ре-импорт внешних матчей стирает фреймы и перезаписывает матч.**
`apps/worker/src/main.ts:618-625` — `match.update` + `matchFrame.deleteMany` +
пересоздание: ручные правки импортированного матча (заметки, статистика)
теряются при следующей синхронизации. Минимум — не перетирать `notes`, если
менялись руками; лучше — поле `manuallyEditedAt` и skip.

### LOW

- **L1.** `serializeExternalMatchNotes` (`worker/main.ts:714`) хардкодит
  `source: 'cuetracker'` — для WST-импорта в notes пишется неверный источник.
- **L2.** Тайминг-энумерация email на login: при несуществующем пользователе
  `auth.service.ts:62` отвечает без вызова argon2 → ответ заметно быстрее.
  Фикс: dummy-verify по фиктивному хэшу.
- **L3.** TOCTOU при регистрации (`auth.service.ts:39-44`): параллельная
  регистрация одного email падает P2002 → 500 вместо 409. Ловить P2002.
- **L4.** `main.ts` — объявления `securityHeaders`/`corsOrigins` вперемешку с
  телом `bootstrap` (работает через hoisting, но читается как обрыв функции).
  Вынести наружу.
- **L5.** AI-очередь: `removeOnFail: false` (`ai.service.ts:277`) — упавшие
  джобы копятся в Redis бессрочно. Поставить `removeOnFail: { count: 500 }`.
- **L6.** `getDecidingFrameResult` (`worker/main.ts:708`) выдирает цифры из
  format: «race to 3» трактуется как best-of-3 — неверная логика декайдера.
- **L7.** nginx `client_max_body_size 200m` против реальных 15 MB API — ужать.
- **L8.** `minio/minio:latest` без пина версии и без healthcheck
  (`docker-compose.prod.yml:142`).
- **L9.** Контейнеры работают от root, без resource limits (mem/cpu) — добавить
  `user:` и `deploy.resources` хотя бы для api/worker.

## 2. Производительность

Состояние в целом хорошее: дашборд ограничен 28 днями, konva грузится через
`next/dynamic`, у TanStack Query `staleTime: 30s`, индексы покрывают основные
паттерны запросов, standalone-сборка Next включена.

- **P1 (= M1).** Отсутствие серверной пагинации — главный будущий деградатор:
  `/matches`, `/training-sessions` тянут полные объекты (включая JSON-поля
  layout-снапшотов) на каждый заход на страницу.
- **P2.** Гигантские клиентские монолиты: `CalendarFactorsClient.tsx` (58 KB),
  `MatchLogClient.tsx` (54 KB), `TrainingSessionClient.tsx` (47 KB) — любой
  setState ре-рендерит весь экран; список+детали+форма стоит разнести на
  мемоизированные подкомпоненты. Это же ускорит работу с кодом.
- **P3.** Внешний импорт делает по 2–4 запроса к БД на каждый матч в цикле
  (`worker/main.ts:577-635`) — при сотнях матчей CueTracker станет медленным;
  батчить (createMany/обновления транзакцией).
- **P4.** Тренировочный список (50 сессий с executions) рендерится целиком в
  левом рейле без виртуализации — на мобильном это километровый DOM (см. UX-3).

## 3. Дизайн / UX (по скриншотам и коду)

Сильные стороны: целостная тёмная система (baize-green акценты), Coach Radar —
отличная идея с confidence-бейджами и action-кнопками, формы с тултипами и
хинтами, fullscreen-скорер, продуманный empty-state.

- **UX-1. Формат дат не локализован.** В форме «Новый матч» нативный
  `mm/dd/yyyy --:--` в русском интерфейсе. Заменить на локализованный
  date-picker (или хотя бы `lang`-атрибут/кастомный контрол). Касается всех форм
  с датами.
- **UX-2. Счёт фреймов «77:27»** читается как время. Использовать тире: 77–27.
- **UX-3. Тренировки на мобильном: список сессий доминирует.** На скриншоте
  mobile видно ~50 плоских карточек до рабочей зоны. Нужны: группировка по
  неделям/месяцам, поиск, сворачивание списка в bottom-sheet на `< lg` (как
  сделано фильтром на /matches).
- **UX-4. Coach Radar многословен**: четыре карточки по 3 абзаца. Сократить до
  «метрика + один вывод», детали — по тапу (progressive disclosure).
- **UX-5. Смешение языков**: системные значения типа «Practice Match» в RU-UI.
  `drill-localization.ts` уже есть — расширить на типы матчей/категории.
- **UX-6. Правый столбец «Новый матч»** — очень длинная форма со
  статистическими полями. Поля Long pot %, Safety %, ошибки и т.п. убрать за
  «Расширенная статистика» (collapsed), оставив счёт/соперника/дату/формат.
- **UX-7.** Логотип/фавикон в корне `C:\work\repos\snooker` по ~900 KB — если
  такие же лежат в `apps/web/public`, сжать (WebP/оптимизированный PNG).

## 4. Что добавить / доработать (взгляд тренера + roadmap ТЗ)

Приоритет 1 — закрывает реальный текущий workflow:

1. **Импорт тренерских заметок из .docx прямо в приложение.** Сейчас отчёты
   Юлиана/Милены вносятся вручную скриптами (`statistics/`, create-milena.ts).
   Сделать: загрузка docx → парсинг брейков → предпросмотр маппинга → создание
   сессий/попыток. Вся логика маппинга уже описана в IMPORT_REPORT.md.
2. **Серверная пагинация + поиск по матчам/сессиям** (см. M1) — без этого
   реальные данные за сезон не помещаются.
3. **Таймер в drill execution.** Упражнения типа «Клиренс цветных по таймеру»
   есть, а секундомера/таймера в панели попыток нет — большой quick win для
   планшета у стола.
4. **Журнал брейков и личные рекорды**: отдельная лента сенчури/полтинников с
   milestone-бейджами (gold-токен бренда как раз для PB). Тренеры в docx ведут
   именно списки брейков.

Приоритет 2 — из roadmap ТЗ (Phase 2), наибольшая ценность:

5. **Skill map** — радар по категориям дриллов (potting / safety / break
   building...), данные уже есть в DrillAttempt.
6. **Тренировочные планы**: тренер назначает недельный план (набор дриллов с
   целевыми метриками), игрок отмечает выполнение; сейчас сессии только ad-hoc.
7. **Комментарии тренера к сессии/матчу** в кабинете (membership уже есть, а
   канала обратной связи нет).
8. **Offline/PWA для drill execution** (ТЗ §15) — у стола Wi-Fi нестабилен;
   очередь мутаций + offline-бейдж заложены в гайдлайны, но не реализованы.
9. **Экспорт отчёта в PDF** (для родителей/федерации) и e-mail-дайджест
   недельного AI-отчёта.
10. **Сравнение периодов** в аналитике («этот месяц vs прошлый») — сейчас
    только фиксированные 28 дней.

## 5. Что сделано хорошо (не трогать)

- Auth: argon2id, ротация refresh с защитой от гонки на стороне БД
  (`updateMany`-CAS), httpOnly+sameSite=strict cookie узкого path, access-токен
  только в памяти, анти-энумерация на resend-verification.
- Авторизация: модель кабинетов (owner/membership, EDIT/VIEW, wellness-гейт,
  `OwnerOnly`/`WriteAccess` декораторы), скоупинг всех выборок по profileId.
- Throttling глобально (120/мин) + точечные лимиты на auth/AI.
- Prisma-схема: индексы по всем FK и фильтруемым полям, уникальные констрейнты
  на токены/фреймы/попытки.
- Деплой: бэкап БД до миграций с abort-on-fail, retention 14, healthchecks
  всех сервисов, bind только на 127.0.0.1, секреты в `.env.deploy` вне git.
- Web: konva через dynamic import, markdown-рендер без innerHTML (XSS-безопасен),
  zustand persist с partialize и миграцией от старых версий.
